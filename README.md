# AI 量化研究助手

这是一个面向期货 CTA 策略研究的本地化工作台。它不是为了做一个通用聊天机器人，也不是为了先搭一个漂亮平台，而是为了服务量化研究员自己的日常研究：更快理解策略、更稳复现规则、更透明地检查交易、更克制地判断策略是否值得继续投入。

当前项目仍处于研究原型阶段。它适合用于策略文章复现、策略初筛、回测证据整理、逐笔交易复核和研究汇报，不应用于直接实盘决策。

## 当前定位

平台要回答的核心问题不是“这个策略看起来好不好”，而是：

- 我是否真正理解了这篇策略文章？
- 平台有没有把原文规则忠实翻译成可运行代码？
- 回测交易是不是由固定引擎真实计算出来，而不是 AI 编出来？
- 每一笔关键交易能不能解释清楚：为什么开仓、何时成交、手数怎么算、为什么退出、盈亏如何得到？
- 这个策略是否值得继续深入研究，还是应该及时放弃？

长期目标是把策略研究做得更简单、更规范、更可验证，而不是让研究员被一堆文本、代码和图表拖累。

## 五阶段工作流

### 阶段一：策略构思

阶段一的目标是让研究员真正看懂策略，而不是简单改写原文。

输入可以是策略文章、交易想法或研究笔记。平台会输出一份研究员可读的策略说明，并同步生成 `strategy_spec.json` 给阶段二使用。

阶段一应重点呈现：

- 策略核心思想。
- 关键概念解释。
- 指标和参数的含义。
- 开仓逻辑、出场逻辑、止损逻辑和手数逻辑。
- 原文模糊但工程上必须落地的默认口径。
- 必须由研究员确认、否则不能进入下一阶段的问题。

后续打磨方向：阶段一应加入更直观的图文解释，例如用小型虚拟 K 线示意图解释信号触发、均线关系、突破条件和止损移动过程。

### 阶段二：建模

阶段二是“策略说明 -> 可执行代码”的翻译层。这里不鼓励 AI 自由发挥，也不希望生成一大段难以阅读的解释。

当前原则：

- 阶段二必须依赖阶段一生成的 `strategy_spec.json`。
- 优先使用受控代码生成器和本地知识库模板。
- 策略代码只负责计算指标和输出信号。
- 持仓、成交、手续费、滑点、保证金、交易记录由固定回测引擎统一处理。
- 代码块需要有简短批注，让研究员能看懂关键逻辑对应阶段一的哪一部分。

策略代码接口固定为：

```text
compute_indicators(df) -> DataFrame
generate_signals(df) -> DataFrame
run(df) -> DataFrame
```

信号约定：

```text
signal_raw =  1：产生多头方向交易信号
signal_raw = -1：产生空头方向交易信号
signal_raw =  0：无新方向信号

exit_signal = 1：只平当前持仓，不反手
exit_signal = 0：不触发只平仓动作
```

后续打磨方向：继续扩充策略模板和 RAG 知识库，减少阶段二对大模型自由生成代码的依赖。

### 阶段三：回测

阶段三由固定回测引擎执行，不由 AI 编造收益、回撤或交易记录。

当前默认口径：

- 日线数据。
- 近五年窗口。
- AKShare / 新浪主力连续合约。
- 单品种验证优先，多品种初筛作为辅助。
- 支持手续费、滑点、保证金、资金不足拒单、涨跌停阻断、低成交量跳过和策略动态止损。

当前输出包括：

```text
reports/web/{strategy}_{symbol}_backtest.html
reports/web/{strategy}_{symbol}_trades.csv
reports/web/{strategy}_{symbol}_equity.csv
reports/web/{strategy}_{symbol}_bars.csv
reports/web/{strategy}_{symbol}_stats.json
```

后续打磨方向：

- K 线图要更像研究工具，而不是普通报告图。
- 均线、通道、止损线、入场点、出场点必须清楚。
- 支持放大、拖动、定位到某笔交易。
- 鼠标移动到线条时应知道它是什么指标。
- 数据口径要升级到可审计的移仓换月数据，记录换月日期、展期价差和换月滑点。

### 阶段四：检查

阶段四是本项目最重要的可信度环节之一。它不是泛泛地评价策略，而是做交易复核。

阶段四应回答：

- 阶段二代码是否忠实复现阶段一？
- 是否存在未来函数？
- 入场、出场、止损、加仓是否和策略说明一致？
- 成交时点是否正确，例如收盘确认、下一根 K 线开盘成交？
- 手数是否按规则计算？
- 手续费、滑点、保证金、净盈亏是否能复算？
- 极端情况如何处理，例如资金不足、涨跌停、低成交量、期末强平？

当前阶段四会抽取典型交易，例如首笔交易、最大盈利交易和最大亏损交易，展示信号、成交、成本和盈亏计算。

后续打磨方向：阶段四应结合 K 线图展示三笔典型交易，让研究员能直接对照信号位置、成交位置、止损线和最终退出原因。

### 阶段五：分析

阶段五只基于真实回测结果和阶段四检查做研究判断。

它要回答的不是“写一份漂亮报告”，而是：

- 策略有没有必要继续深入？
- 收益来自稳定规则，还是来自少数行情？
- 当前版本的主要风险是什么？
- 如果继续优化，应该先改参数、换品种、改数据口径，还是调整策略结构？
- 进入模拟盘或实盘前还缺什么证据？

阶段五会保留策略评分卡，但评分不是目的。评分的作用是帮助研究员快速定位问题：收益、回撤、交易质量、稳定性、实盘可行性和证据完整性。

## 样本策略

当前保留一个完整历史样本策略：

- `locked_vwma_obv`：简单有效的均线量化策略。

该样本用于展示完整五阶段结果。当前结论是：它可以作为 v1 基线继续研究，但不能直接推导为可实盘策略。

样本策略后续应继续打磨：

- 更准确解释锁定均线和 6 个条件变量。
- 标记原文模糊处和平台默认复现口径。
- 检查动态移动止损是否过早截断趋势。
- 做多品种、多参数和移仓换月数据验证。

## 当前能力边界

比较适合：

- 均线类策略。
- 突破类策略。
- 海龟/唐奇安通道类策略。
- ATR 止损类策略。
- OBV、VWMA、斜率等趋势过滤策略。

仍需谨慎：

- 多周期联动。
- 多品种组合。
- 套利和价差策略。
- 复杂加减仓状态机。
- 盘中高频条件。
- 依赖真实主力换月、成交约束和盘口微观结构的策略。

## 本地运行

启动后端：

```powershell
python -X utf8 -m uvicorn app.main:app --host 127.0.0.1 --port 8015
```

浏览器访问：

```text
http://127.0.0.1:8015/
```

首次使用前复制配置模板：

```powershell
copy app\config.example.yaml app\config.yaml
```

然后在页面设置 API Key。

## 测试

```powershell
python -X utf8 -m unittest tests.test_platform_contracts
```

## 安全说明

`app/config.yaml`、本地行情数据、回测报告、草稿策略、外部技能包和临时文件默认不进入 GitHub。不要提交真实 API Key、交易账号、密码或客户数据。
