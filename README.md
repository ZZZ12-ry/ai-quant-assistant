# AI 量化研究助手

面向期货 CTA 研究员的阶段化 AI 研究工作台。  
系统围绕“策略构思 - 建模 - 回测 - 检查 - 分析”组织研究流程，把原本分散在文章阅读、规则拆解、代码实现、回测验证、交易复核和研究归档中的工作，收敛为一条连续、可追踪、可回看的研究链路。

一句更像产品口号的表达是：

> 让 AI 陪伴策略从想法、建模与验证，一直走到研究归档。

---

## 项目定位

这个项目不是一个普通聊天工具，也不只是一个回测页面。  
它的定位是：

- 帮助研究员更快看懂一篇策略
- 帮助研究员更稳地把策略原文落成规则和代码
- 帮助研究员更透明地检查每一笔关键交易
- 帮助研究员更有依据地判断一个策略是否值得继续深入
- 帮助研究员把策略作为可迭代的研究对象持续验证，而不是一次性产出

它的核心价值不在“自动生成了多少内容”，而在于把策略研究这件事做得更：

- 有结构
- 有证据
- 有上下文
- 有历史沉淀
- 有复核能力

---

## 这个项目设计上最好的地方

这个项目最重要的设计，不是某个单点功能，而是它把策略研究拆成了符合研究员思维习惯的五个阶段，并把“研究结论 -> 版本迭代 -> 再验证”做成了闭环：

1. 策略构思
2. 建模
3. 回测
4. 检查
5. 分析

这意味着研究员不需要在“看文章、写代码、跑回测、做解释、做汇报”之间来回切换，而是可以沿着一个固定流程推进，并且在每个阶段都留下明确产物。

这种阶段化设计有几个明显优势：

- 每一步都知道当前在解决什么问题
- 上下文不会轻易丢失
- 结果可以逐阶段沉淀，而不是只留一段聊天记录
- 一个策略完成后，可以作为完整样本反复回看
- 后续做策略迭代时，有清晰的基线版本可对照

从产品设计角度看，这不是“把 AI 接进来”，而是把 AI 放进了一个真正服务研究工作的流程里，并且让每轮研究都能自然进入下一轮实验。

---

## 五阶段研究工作流

### 1. 策略构思

目标不是复述原文，而是让研究员真正看懂策略。

这一阶段会围绕原始策略文章、研究笔记或交易想法，整理出：

- 策略核心思想
- 关键概念解释
- 指标与参数含义
- 开仓、出场、止损、加仓、手数逻辑
- 原文模糊但工程上必须落地的默认口径
- 必须人工确认的问题

阶段一的核心价值，是把“模糊的自然语言策略描述”变成“研究员和系统都能继续使用的结构化策略说明”。

同时，系统会生成结构化策略规格文件，例如：

```text
strategy_spec.json
```

供后续阶段继续使用。

---

### 2. 建模

目标是把策略说明稳定翻译成可执行代码，而不是让模型自由发挥。

这一阶段强调：

- 代码生成受约束
- 依赖结构化 spec 和本地模板
- 代码逻辑和阶段一说明一一对应
- 策略代码只负责指标与信号
- 持仓、撮合、成本、记录由引擎统一处理

当前固定接口为：

```text
compute_indicators(df) -> DataFrame
generate_signals(df) -> DataFrame
run(df) -> DataFrame
```

信号约定：

```text
signal_raw =  1   新多头信号
signal_raw = -1   新空头信号
signal_raw =  0   无新方向信号

exit_signal = 1   只平当前持仓，不反手
exit_signal = 0   不触发只平仓动作
```

当前已补充的模板能力包括：

- 均线 / 通道 / 趋势跟踪类策略
- OBV / VWMA / 斜率过滤类策略
- ATR 止损类策略
- 主动止盈 / 分批止盈类策略

---

### 3. 回测

目标是生成真实可复核的交易证据，而不是只输出几个漂亮指标。

当前默认口径：

- 日线数据
- 近五年窗口
- AKShare / 新浪主力连续合约
- 单品种验证优先，多品种初筛作为辅助
- 支持手续费、滑点、保证金、资金不足拒单、涨跌停阻断、低成交量跳过、动态止损等执行细节

当前输出包括：

```text
reports/web/{strategy}_{symbol}_backtest.html
reports/web/{strategy}_{symbol}_trades.csv
reports/web/{strategy}_{symbol}_equity.csv
reports/web/{strategy}_{symbol}_bars.csv
reports/web/{strategy}_{symbol}_stats.json
```

当前图表设计方向是“研究工具化”：

- 主图只保留价格同量纲信息
- 非价格指标按当前策略实际产出动态展示
- 入场点、出场点、止损线与关键执行线在同一图层可对照
- 权益与回撤独立展示
- 图内大段说明性标题已移除，优先保证图表可读性

---

### 4. 检查

这是系统可信度最重要的部分之一。

阶段四不是泛泛评价策略，而是做“交易复核”。

重点检查：

- 阶段二代码是否忠实复现阶段一
- 是否存在未来函数
- 入场、出场、止损、加仓是否和说明一致
- 成交时点是否正确
- 手数是否按规则计算
- 手续费、滑点、保证金、净盈亏是否能复算
- 极端情况如何处理

当前会抽取典型交易，例如：

- 首笔交易
- 最大盈利交易
- 最大亏损交易

并展示相关信号、成交、成本和盈亏计算线索。

---

### 5. 分析

目标不是生成一份“看起来像报告”的文本，而是给出研究判断。

阶段五要回答的问题包括：

- 这个策略有没有继续深入的必要
- 收益来自稳定规则还是少数行情
- 当前版本最主要的风险点是什么
- 下一轮优化应该优先改参数、换品种、调数据口径还是改结构
- 距离模拟盘 / 实盘还差哪些证据

系统会保留评分卡，但评分不是目的。  
评分的作用是帮助研究员快速定位问题，例如：

- 收益质量
- 回撤压力
- 交易质量
- 稳定性
- 实盘可行性
- 证据完整性

## 历史策略与研究沉淀

当前界面设计上，历史区主要展示正式保存的历史策略，而不是把所有草稿都暴露给研究员。

这样做的好处是：

- 历史列表更干净
- 已完成策略更像正式研究资产
- 每个正式策略都可以作为后续迭代的基线版本

当一个策略完整走完流程并保存后，它会作为正式版本进入历史回看体系，用于：

- 查看五阶段完整产物
- 对照不同策略
- 作为后续迭代研究的母版本

---

## 策略版本与迭代闭环

这个项目不是“做完五阶段，保存一份报告就结束”。  
在当前实现里，策略天然被视为一个可以持续迭代、持续验证的研究对象。

完成一轮五阶段之后，研究员可以直接基于当前完成版本创建新的迭代草稿，而不是覆盖旧版本。

当前已经实现的能力包括：

- 从已完成策略创建迭代版本
- 从已有迭代版本继续分支出下一个版本
- 保留父策略、父版本和版本作用域关系
- 记录本次迭代类型、修改说明和研究假设
- 将新版本作为独立草稿进入下一轮五阶段流程
- 自动清除已经失效的下游回测、检查和分析产物
- 在版本完成后对比父版本与当前版本的关键指标
- 在历史策略中按策略和版本组织结果，而不是平铺保存

这意味着平台实际支持的是这样一条研究路径：

```text
完成策略 v1
    ↓
阶段五提出问题或改进方向
    ↓
创建迭代版本
    ↓
记录修改假设与验收标准
    ↓
重新检查策略说明、代码和参数
    ↓
重新回测、复核和分析
    ↓
与父版本比较
    ↓
保留、继续迭代或放弃
```

这部分能力让它不只是一个五阶段工作台，也更接近一个轻量的策略实验管理系统。

### 当前迭代设计的价值

- 原策略不会被覆盖，研究证据可以回看
- 父子版本关系明确，方便追踪每次修改的原因和结果
- 无效的旧回测和旧分析不会污染新版本
- 迭代不仅记录“改了什么”，也记录“为什么改”
- 回测完成后可以直接比较改动是否真的提升了策略证据

对量化研究来说，这比单纯保存历史报告更有价值，因为它开始具备实验管理、数据血缘和研究闭环的意味。

## 样本策略

当前保留了一个完整历史样本策略：

- `locked_vwma_obv`：简单有效的均线量化策略

这个样本的意义不只是“展示页面”，而是作为完整研究样本存在：

- 它展示了五阶段如何贯通
- 展示了策略说明如何沉淀为代码、回测、检查与分析
- 为后续策略迭代提供了一个可比较的基线

---

## 当前能力边界

当前更适合处理的策略类型：

- 均线类策略
- 突破类策略
- 海龟 / 唐奇安通道类策略
- ATR 止损类策略
- OBV、VWMA、斜率等趋势过滤策略

当前仍需谨慎对待：

- 多周期联动策略
- 多品种组合策略
- 套利 / 价差策略
- 复杂状态机加减仓逻辑
- 高频盘中条件
- 强依赖真实主力换月、盘口微观结构和成交约束的策略

## 为什么这个项目有继续做下去的价值

这个项目真正有价值的地方，不在于“它是不是已经是完美平台”，而在于它已经形成了一个很好的研究产品雏形：

- 它有清晰的阶段化工作流
- 它有面向研究员的结构化输出目标
- 它有策略说明、代码、回测、检查、分析之间的贯通关系
- 它有历史策略沉淀与回看机制
- 它已经不再只是“AI聊天”，而是在服务策略研究流程本身

换句话说，它已经证明了一件重要的事：

> AI 不只是用来回答问题，还可以用来组织和放大策略研究工作。

这就是这个项目最好的设计价值。
